# 需求理解与假设说明

本文档对应考点 6，逐项说明题目中未给出精确数值的留白规则，给出具体假设值及其依据。

---

## ① 分级审批金额阈值

**假设值**：二级审批金额阈值 = 5,000 元（CNY）

**依据**：
- 参考物流行业常见客单价区间（200-5000 元），5000 元是中等偏上的运单金额
- 低于 5000 元的异常可由一级审批人（如仓库主管）直接处理，无需升级
- 高于 5000 元的异常涉及较大金额赔付，需二级审批（如区域经理）把关
- 阈值可在后台 `app_config` 表中动态调整，不硬编码在代码中

---

## ② 审批超时时长

**假设值**：
- 待审批（pending）超时：24 小时 → 自动升级二级审批
- 一级审批超时：48 小时 → 自动升级二级审批
- 二级审批超时：72 小时 → 自动驳回

**依据**：
- 待审批阶段是工单创建后等待分配，24 小时内应有审批人接单处理
- 一级审批 48 小时：覆盖跨工作日场景（周五下午创建的工单，周一中午前处理）
- 二级审批 72 小时：涉及更高层级审批人，给予更充裕时间
- 二级审批超时采用"自动驳回"而非"自动升级"：已经是最高审批层级，无法继续升级；驳回后允许重新提交仍有一次完整的审批机会
- 超时时长可在后台 `app_config` 表中配置调整

---

## ③ 重新提交次数上限

**假设值**：最多 3 次重提，超限后自动关闭工单（rejected_final）

**依据**：
- 3 次是常见的审批重试上限（如常见的采购审批系统、OA 系统）
- 每次拒绝意味着审批人发现了必须修正的问题，反复提交说明上报人或审批人之间存在共识缺口
- 超过 3 次后自动关闭而非强制升级：此时问题已不是审批层级不足，而是上报信息本身有问题，需重新走完整流程
- 采用 `auto_close` 而非 `auto_escalate`：因为反复被拒说明业务数据本身有争议，强制升级不能解决根本问题

---

## ④ 物流异常类型映射（各类型对应的下游动作）

| 异常类型 | 下游动作 | 是否赔付 | 赔付方向 | 库存影响 |
|----------|----------|----------|----------|----------|
| 丢件 (lost) | 理赔 + 重新发货 | 是 | 赔付客户 | 减库存 |
| 破损 (damaged) | 理赔 + 退货入库 | 是 | 赔付客户 | 增库存 |
| 客户拒收 (rejected) | 退货入库 | 否 | — | 增库存 |
| 超时未签收 (timeout) | 重新发货 | 否 | — | 减库存 |
| 地址错误 (address_error) | 重新发货 | 否 | — | 减库存 |

**依据**：
- 丢件和破损涉及货品损失，属于物流服务商/承运方责任，需要赔付客户
- 客户拒收通常不涉及赔付（客户主观原因），但需要退货入库
- 超时未签收和地址错误本质上是物流调度问题，重新发货即可，一般不涉及赔付
- 地址错误也可能是客户自身原因，赔付不合理

### 品控类异常映射

| 异常类型 | 下游动作 | 是否赔付 | 赔付方向 |
|----------|----------|----------|----------|
| 数量不符 | 退回供应商 + 重新采购 | 是 | 向供应商追偿 |
| 外观破损 | 退回供应商 + 重新采购 | 是 | 向供应商追偿 |
| 规格不符 | 退回供应商 + 重新采购 | 是 | 向供应商追偿 |
| 标签错误 | 退回供应商 | 是 | 向供应商追偿 |
| 批次异常 | 退回供应商 + 重新采购 | 是 | 向供应商追偿 |

品控异常与物流异常的核心区别是赔付方向：品控异常问题出在供应商端，赔付方向是"向供应商追偿"；物流异常问题出在物流环节，赔付方向是"赔付客户"。

## ⑤ 角色权限划分

| 角色 | 权限 | 可审批 | 可快速放行 | 限制 |
|------|------|--------|------------|------|
| 操作员 (operator) | 扫描录入、异常上报、查看工单 | 否 | 否 | 不能审批任何工单 |
| 品控主管 (qc_supervisor) | +品控管理、快速放行、规则配置 | 否 | 是 | 不能审批工单、不能对自己操作的扫描做快速放行 |
| 一级审批人 (level1_approver) | 查看工单、一级审批 | 是（金额≤阈值） | 否 | 不能审批自己上报的工单 |
| 二级审批人 (level2_approver) | 查看工单、二级审批 | 是（所有层级） | 否 | 不能审批自己上报的工单 |
| 管理员 (admin) | 全部权限 | 是 | 是 | 无 |

### 权限边界说明
- **上报人不能审批自己提交的工单**：后端接口有校验（比对 reporter 和 approver 字段）
- **品控主管与审批人权限不重叠**：品控主管专注品控质量、审批人专注审批流程，职责分离
- **一人可兼任多个角色**：系统允许同一人同时具备品控主管和审批人角色，但有操作限制（如不能对自己操作的扫描做快速放行）
- **非对应层级的审批人不能越权操作**：后端接口根据 `approverRole` 校验审批权限层级

### 归属校验（单租户假设）
当前 V2 系统数据不区分租户（单租户假设）。如需支持多租户：
- 在 `exception_tickets` 表增加 `tenant_id` 字段
- 在 `MOCK_USERS` 中增加 `tenant_id`
- 所有查询添加 `tenant_id` 过滤条件
- V2 接口增加租户上下文传递

## ⑥ V2 数据同步频率与一致性策略

**策略**：按需实时拉取 + 本地缓存

| 场景 | 策略 |
|------|------|
| 异常上报（发起时） | 实时调用 V2 接口校验运单存在性，最新数据写入本地快照 |
| 扫描录入 | 实时调用 V2 接口校验 SKU 归属 |
| 工单详情展示 | 使用本地快照数据，标注来源和同步时间 |
| 工单列表展示 | 使用本地快照数据 |
| V2 不可用 | 使用过期缓存数据，前端标注"数据可能非最新，获取自 XX 时间" |

**缓存有效期**：1 小时。超过 1 小时的缓存数据视为"过期"，前端加注"数据可能非最新"。

**一致性策略**：
- 关键操作（异常上报、扫描 SKU 校验）做实时接口校验，不依赖缓存
- 展示类操作使用缓存以减少 V2 接口压力
- "V2 数据在异常处理期间发生变更"的处理：采用"实时校验"方式。
  在审批关键节点（如审批通过后执行前）再次校验一次运单金额，若发现不一致则以 V2 最新值为准并通知审批人。若差异过大（>20%）则暂停执行，标记为"需人工确认"。

## ⑦ 品控暂扣超时时长

**假设值**：2 小时（独立于审批超时的 24/48/72 小时）

**依据**：
- 品控暂扣意味着货物物理上被锁在仓库，产生压仓成本（仓储费、周转率下降）
- 2 小时足够品控主管到现场复核扫描结果，做出"确认异常"或"误判放行"决定
- 审批超时（24-72h）以天为单位，品控暂扣超时必须以小时为单位，否则货物压仓 2-3 天不可接受
- 超时后处理方式：品控暂扣超时直接进入二级审批（level2_review），而非待审批状态（pending）
  - 理由：品控异常已经过品控主管复核（2h），到二级审批阶段说明异常属实需要高层决策

## ⑧ 品控规则触发阈值

| 规则名称 | 条件字段 | 操作符 | 阈值 | 严重度 | 审批层级 |
|----------|----------|--------|------|--------|----------|
| 数量差异检测 | quantity_diff_percent | > | 5% | 高 | 一级 |
| 严重数量差异 | quantity_diff_percent | > | 20% | 严重 | 二级 |
| 外观破损-轻微 | damage_level | ≥ | 1 | 低 | 一级 |
| 外观破损-严重 | damage_level | ≥ | 3 | 严重 | 二级 |
| 规格偏差检测 | spec_deviation | > | 0 | 中 | 一级 |
| 标签错误检测 | label_match | = | false | 中 | 一级 |
| 批次异常检测 | batch_valid | = | false | 高 | 二级 |

**依据**：
- **数量差异 5%**：参考零售行业常见的商品损耗率（1-5%），超过 5% 属于异常
- **数量差异 20%**：严重差异，可能是整箱缺失，需要供应商端查因并追偿
- **破损等级**：采用 5 级制（0=完好, 1-2=轻微, 3-4=明显, 5=完全损毁），≥1 触发注意，≥3 触发严重处理
- **规格偏差**：原则上零容忍，检测到即触发
- **标签/批次**：异常属于合规问题，一旦发现必须处理
- 所有阈值均为**可配置项**，存储在 `qc_rules` 表中，后台可直接调整

## ⑨ 品控主管角色权限边界

| 权限项 | 是否允许 | 说明 |
|--------|----------|------|
| 执行品控扫描 | 是 | 品控主管可直接进行扫描操作 |
| 查看所有品控工单 | 是 | 全局视角 |
| 误判快速放行 | 是 | **仅品控主管可操作**，需填写复核原因，操作留痕 |
| 配置品控规则 | 是 | 可增删品控规则 |
| 审批工单（一级/二级） | **否** | 品控与审批职责分离 |
| 快速放行自己的扫描 | **否** | 不能自己查自己 |
| 兼任审批人 | 允许但有限制 | 对同一工单只能行使一种角色职能 |

### 快速放行操作规范
1. 仅品控主管或管理员可操作
2. 必须填写复核原因（不允许空原因）
3. 操作记录写入审批记录表（approval_records），action=approve, opinion 标注"【误判快速放行】"
4. 放行后同步解锁扫描批次状态和库存状态
5. 仅适用于扫描触发的品控工单（exceptionSource=scan_trigger）

## 如果能问产品经理的问题清单

以下问题体现主动澄清意识：

1. "品控暂扣超时 2 小时后，是自动进入二级审批还是有其他流程？当前设计是进入二级审批，是否符合业务预期？"
2. "不同仓库/不同品类货物是否可以有不同的品控规则？如果可以，规则表需要增加仓库/品类关联字段。"
3. "同一运单的品控工单未关闭期间，又发生物流丢件，是否允许两份不同类型工单并存？当前设计允许并存。"
4. "赔付金额是固定按运单金额全额赔付，还是需要人工调整？当前设计审批通过后自动按工单金额生成赔付记录。"
5. "审批人离职后工单转交的具体流程：是自动转给同组其他审批人、还是需要管理员手动分配？当前仅实现了'超时自动处理'的机制。"
6. "V2 运单金额字段如果是单价×数量计算得出，V3 拉取时发现金额为 0 如何处理？"
7. "扫描录入是否需要支持批量扫描（一次扫描多个 SKU）？当前设计是单个 SKU 逐一扫描。"
8. "品控主管快速放行后，是否需要在 V2 侧也标记该运单状态？当前未做回写。"
9. "降级处理（降价出库）的差价由谁承担？客户还是供应商？当前设计默认由供应商承担差价。"
10. "是否需要支持工单的优先级排序？（如紧急工单优先处理）"

## 补充：V2 接口兼容性说明

### 如果 V2 原本没有对外接口，新增策略

1. **接口版本策略**：新增 `/api/v2/external/` 路径前缀，与 V2 内部 API 隔离
2. **字段向后兼容**：新增字段必须设为可选（nullable），不修改已有字段类型
3. **灰度上线**：
   - Phase 1：接口部署到 V2 测试环境，V3 对接测试
   - Phase 2：接口部署到 V2 生产环境，V3 切换到生产接口
   - Phase 3：监控错误率，确保不影响 V2 现有调用方
4. **V2 接口变更时 V3 应对**：
   - V3 快照表保存 `raw_data` 字段（JSONB），可容忍 V2 新增字段
   - 若 V2 核心字段类型变更（如 `amount` int → decimal），V3 在读取快照时通过 `data_version` 感知并进行兼容转换
   - 建立接口变更通知机制（如 Webhook），V2 字段变更时通知 V3
